# Plan: Export Device-Level Metrics from Digital Ocean Droplets Instead of Local Docker

## Current State

The local Docker stack (`docker/docker-compose.yml`) runs 4 custom Python mock exporters (ports 9100-9103) that emit per-device labeled metrics. A local Prometheus scrapes them. The backend connects to `http://168.144.91.25:9090` (remote DO Prometheus) by default, but the remote Prometheus only scrapes standard `node-exporter` and `cadvisor` — not the custom device-level metrics.

**Goal**: Deploy the custom per-device mock exporters onto each Digital Ocean droplet, configure the mgmt droplet's Prometheus to scrape them, and remove local mock exporters from the primary Docker stack.

## Key Constraints

- User: "Don't modify or change any codes other than the related codes for this task"
- Port 9100 is already used by `node-exporter` on each droplet → mock exporter must use port **9200**
- Backend `config/settings.py` already defaults `PROMETHEUS_URL=http://168.144.91.25:9090` — no backend change needed

## Changes Required

### 1. Create Remote Mock Exporter (`terraform-pipeline/mock-exporter/`)

Create a single parameterized exporter that accepts `DROPLET_NAME` env var, replacing the 4 separate scripts:

**New file: `terraform-pipeline/mock-exporter/Dockerfile`**
```dockerfile
FROM python:3.12-slim
WORKDIR /app
RUN pip install --no-cache-dir prometheus-client
COPY exporter.py .
ENTRYPOINT ["python", "exporter.py"]
```

**New file: `terraform-pipeline/mock-exporter/exporter.py`**
- Single parameterized script reading `DROPLET_NAME` from env
- Reads device/edge definitions from a JSON config or inline dict keyed by droplet name
- Emits the exact same Prometheus gauge names and label schemas as the current local exporters
- Listens on port 9200

**New file: `terraform-pipeline/mock-exporter/devices.json`**
- JSON mapping of droplet name → list of devices (id, role) and edges (source, target)
- Derived from the 4 existing `mock_exporter_local_droplet{1-4}.py` DEVICES and EDGES lists

### 2. Update Cloud-Init Files (4 files)

Each droplet's cloud-init gets the mock exporter container added to its embedded docker-compose:

**Files to modify:**
- `terraform-pipeline/cloud-init/cloud-init-tor1.yaml`
- `terraform-pipeline/cloud-init/cloud-init-tor2.yaml`
- `terraform-pipeline/cloud-init/cloud-init-mgmt.yaml`
- `terraform-pipeline/cloud-init/cloud-init-storage.yaml`

**For each (except mgmt):** Add to the embedded `docker-compose.yml`:
```yaml
mock-exporter:
  image: <built-image-or-build-locally>
  container_name: mock-exporter
  environment:
    - DROPLET_NAME=<droplet-name>
  ports:
    - "9200:9200"
  restart: unless-stopped
```

**For mgmt cloud-init:** Additionally update the Prometheus `scrape_configs` to add a new job:
```yaml
- job_name: "device-metrics"
  scrape_interval: 12s
  static_configs:
    - targets:
        - "DROPLET_1_PRIVATE_IP:9200"
        - "DROPLET_2_PRIVATE_IP:9200"
        - "DROPLET_3_PRIVATE_IP:9200"
        - "DROPLET_4_PRIVATE_IP:9200"
```

**Image distribution approach**: Two options:
- **Option A (recommended)**: Build the image locally, push to a container registry (e.g., Docker Hub or DigitalOcean Container Registry), reference by tag in cloud-init
- **Option B**: Have cloud-init COPY the exporter script and build the image on-droplet at boot time (simpler, no registry needed, but slower boot)

Remove or comment out the 4 local mock exporter services and the local Prometheus service. Keep Redis, Neo4j, and InfluxDB (they're still needed locally). This makes the local stack lightweight — just data stores, no mock telemetry.

Optionally add a `prometheus.local.yml` note that it's for development/preview fallback only.
